# エージェント 2方式比較メモ（Phase 3 M4）

2026-07-10 実施。同一のエージェントコード（自前 tool use ループ + read-only ツール4種）を、**② ローカル実行**と **① AgentCore Runtime（東京・直接コードデプロイ）**の2方式で動かし、挙動・運用特性を比較した。

## 実験設定

- エージェント: `client/agent_cli.py`（Converse API の toolConfig / toolUse / toolResult を自前で回す。MAX_STEPS=8）
- ツール: get_cost（Cost Explorer）/ search_logs（invocation logging の CWL）/ list_resources（Tagging API）/ search_kb（Phase 2 の KB Retrieve）— **すべて read-only**
- モデル: `jp.anthropic.claude-sonnet-4-6`（両方式共通。エージェントコードが同一なので当然同じ）
- 方式①: 同コードを zip（19MB、arm64/py3.13 依存同梱）で S3 に置き、`aws_bedrockagentcore_agent_runtime`（Terraform）でデプロイ。`@app.entrypoint` の 40 行ラッパのみ追加
- 質問セット: 単一ツール（リソース棚卸し）/ KB 検索 / マルチステップ（コスト→原因調査）+ 絞り込みフォローアップ（「そのうち Bedrock 分だけ詳しく」）

## 2方式の結果

| | ② ローカル (agent_cli.py) | ① AgentCore Runtime |
|---|---|---|
| 実行場所 | 手元の Mac | 東京リージョンの microVM（セッション単位で分離） |
| 認証・ツールの鍵 | 利用者ロール `bedrock-lab-user` | Runtime 実行ロール（**同一ポリシー2枚を付け替えただけ**） |
| 会話文脈 | CLI プロセスのメモリ（起動中のみ） | セッション microVM のメモリ（idle 5分 / 最長30分で消滅） |
| ツール実行の観測 | `--verbose` で手元に表示 | クライアントからは見えない。stdout が CloudWatch に自動転送 |
| 同一質問のレイテンシ（棚卸し） | 18.2s | 14.1s（差は生成の揺れの範囲） |
| コールドスタート | なし | **体感差なし**（アイドル後の初回も 14.0s） |
| デプロイ | 不要（python 実行） | zip ビルド + terraform apply（Docker/ECR 不要） |
| 追加コスト | なし（モデル + CE のみ） | Runtime 実行秒課金（vCPU/メモリ。実測額は CE 反映後に確認） |
| 絞り込みフォローアップ | ✅ steps=0（ツールを呼ばず前回回答から再構成、in=2,989） | ✅ ただし **search_logs を自発追加**してよりリッチに回答（CloudWatch で確認） |

## 発見

### 1. 「エージェント」というリソースは存在しない — 実体はループ + ツールカタログ

Bedrock Agents Classic と違い、AgentCore にはエージェントの「定義」を登録する場所がない。デプロイしたのは**ただの Python コード**で、LLM は毎ターン「次にどのツールを使うべきか」を答えるだけ。実行は常にアプリ側（ローカルなら手元、Runtime ならクラウド側のプロセス）にある。エージェント化とは特別な何かではなく、**「デプロイ・IAM・ログ」のいつもの三点セット**だった。

### 2. 同一コード・同一質問でも、ツール選択は毎回同じにならない

「そのうち Bedrock 分だけ詳しく」に対し、ローカルでは **steps=0**（ツールを一切呼ばず前回回答から再構成、「$0.01 の節約です」と自己申告）、Runtime では **get_cost の再呼び出しは節約しつつ search_logs を自発的に追加**し、ログ考察付きの回答を返した。方式の差ではなく **LLM の非決定性**であり、どちらも合理的だが同じではない。**エージェントの回帰テストは「出力一致」では書けない**——観点評価（ツール呼び出し回数の上限、禁止操作の不在、回答の根拠性）で設計する必要がある。企業導入で必ず問われる性質が2回の実行で実測できた。

### 3. 権限設計は「同じ鍵束の、持ち主の付け替え」

方式②では利用者ロールに、方式①では Runtime 実行ロールに、**同一の IAM ポリシー2枚**（モデル+KB / read-only ツール）をアタッチした。Terraform 上で「鍵束は同じ、持ち主だけ違う」構図がそのまま見える。ローカル実行は「作業者の権限で動く便利スクリプト」、Runtime は「サービスアカウントで動く共用サービス」——インフラ運用で繰り返してきた昇格作業と同型。

### 4. 観測性が逆転する

ローカルの `--verbose`（手元にツール実行が見える）は、Runtime ではクライアントに届かない。代わりに stdout が CloudWatch（`/aws/bedrock-agentcore/runtimes/...`）へ自動転送され、**デバッグ出力がそのまま運用ログになる**。「Runtime が実際に何のツールを呼んだか」はログを見ないと分からない——エージェントの運用では**ログ設計がエージェントの可観測性そのもの**になる。

### 5. セッションの実体は microVM のプロセスメモリ

Runtime はセッション ID ごとに分離された microVM を立て、プロセスが数分生き残る。会話履歴（Python のリスト）がそのメモリに残るため、**アプリコードを変えずにセッション内マルチターンが成立した**。`/new`（別セッション ID）では文脈が消え、CloudWatch でも get_cost の再実行が確認できた。idle timeout（今回 300 秒）が「会話の寿命」を決める運用パラメータになる。コールドスタートは体感できなかった（直接コードデプロイのセッション生成は高速という仕様どおり）。

### 6. レイテンシは今回も生成が支配（Phase 2 発見3の再確認）

同一質問でローカル 18.2s vs Runtime 14.1s——**Runtime 化のオーバーヘッドは生成時間の揺れに沈む**。ツール実行は 121ms〜1.1s（CE が最も遅い）で、体感を決めるのは相変わらずモデルの生成。実行基盤の選択はレイテンシではなく、権限・観測・共用性の要件で決めるべき。

## 方式によらない観察（エージェント一般の挙動）

- **ツール結果の批判的解釈**: 「ERROR」フィルタのヒットが「埋め込み処理に流れた記事本文内の文字列」であることを見抜き、誤検知と正しく報告した
- **read-only の線引き**: 「FSx を削除して」に対し、照会専用である旨を説明して手順だけ案内（steps=0）。設計どおり
- **限界の自己申告**: 呼び出し回数の集計で「max_events 上限により実際はもっと多い可能性、正確には Logs Insights を」と自分のツールの不足を明示した
- **KB 空振り時に再検索しない**: 正解が KB にある質問（S3 Vectors の注意点）で、抽象的な検索クエリが空振りすると「記録がない」と正直に回答した（ハルシネーションなし）が、**語彙を変えた再検索は試みなかった**（ステップ余力あり）。システムプロンプトで再検索を促す改善余地。エラー文言起点のクエリでは同じ内容を正しく引けており、Phase 2 発見2（短い抽象クエリは検索に弱い）の再確認でもある
- **入力トークンの構造**: ツール往復を履歴に積まない設計でも、ツール結果が大きいターンは in が膨らむ（ログ50件取得で in=17,350）。ツール定義+システムプロンプトの毎ターン再送と合わせ、**Phase 4（プロンプトキャッシュ）の効果測定ベースライン**になる

## 使い分け指針（個人ラボの結論）

- **開発・実験・自分だけが使う** → ② ローカル（デプロイ不要、`--verbose` で挙動が見える）
- **共用・常設・権限を作業者から分離したい** → ① Runtime（API 化・実行ロール・CloudWatch。ただし挙動の観測はログ設計次第）
- 実装コストの差は小さい（ラッパ 60 行弱 + zip ビルド + Terraform ~100 行）。**同じコードで両対応にしておき、用途で選ぶ**のが現実解

## 再現方法

```bash
# ② ローカル
python agent_cli.py --profile bedrock-lab --verbose

# ① Runtime (デプロイ)
scripts/build_runtime_zip.sh && (cd terraform && terraform apply)

# ① Runtime (呼び出し)
python invoke_runtime.py --profile bedrock-lab --verbose

# Runtime 側のツール実行を確認
aws logs filter-log-events \
  --log-group-name "/aws/bedrock-agentcore/runtimes/<runtime-id>-DEFAULT" \
  --filter-pattern '"[step"' --region ap-northeast-1
```

計測ログは `client/logs/agent_*.jsonl`（steps / usage を記録）。
