# RAG 3方式比較メモ（Phase 2 M4）

2026-07-09 実施。同一の Knowledge Base（Qiita 記事35本 + lab-docs 3本、Titan V2 / 1024次元 / S3 Vectors）に対して、3つの実装方式で同一の2ターン会話を実行し、挙動・性能・コストを比較した。

## 実験設定

- **Q1**: 「ALB の 502 エラーの切り分け方法は？」（初回質問）
- **Q2**: 「その3層のうち Target Group 層だけ詳しく」（**絞り込みフォローアップ** — 会話文脈の扱いを試すのが狙い)
- 生成モデルは全方式 `jp.anthropic.claude-sonnet-4-6`、検索は top_k=5
- 実装: 方式① = `chat.py --rag` / 方式②③ = `mini_rag.py --search kb|vectors`

## 3方式の結果

|---|---|---|---|
| 実装 | フルマネージド | 検索のみ managed、合成・履歴は自前 | 埋め込み→検索→合成すべて自前 |
| **Q2 絞り込み追従** | ❌ 3層全体を再説明 | ✅ Target Group 層のみ深掘り | ✅ Target Group 層のみ深掘り |

| 生成レイテンシ (Q1) | (未計測) | 22.6s | 12.1s |
| 入力トークン (Q1) | 不明 (内部) | **6,081** | **1,820** |
| 返却チャンク実測 | 不明 (内部) | 3件・最大 **8,120字** (結合/親チャンク相当) | 5件・228〜690字 (生チャンク) |

## 発見

### 1. フォローアップの絞り込みは「会話履歴を誰が持つか」で決まる

①は毎ターン検索が走り、生成が検索チャンクに支配されるため「〜だけ詳しく」に追従できない（3層を律儀に再説明)。②③は履歴を自前で持ち、システムプロンプトで「重複回避・絞り込み追従」を指示できるため、期待どおり動いた。**手軽さの①、制御性の②③**。

### 2. フォローアップ時、検索はほぼ役に立っていない

Q2 の検索スコアは全方式で激減（③: 0.77→0.37 / ②: 0.88→0.68。無関係な記事もヒット)。短い絞り込みクエリは検索クエリとして弱い。それでも回答が正確なのは**会話履歴が補っている**から。マルチターン RAG では「毎ターン素朴に検索」は無駄が多く、クエリ書き換え（会話文脈を含めた検索クエリ生成）や検索スキップ判定に改善余地がある。

### 3. レイテンシの98%は生成

検索は 200〜500ms、生成は 9〜23s。RAG の体感速度を決めるのは検索基盤ではなく**生成側**（モデル選択・出力長・ストリーミング表示)。S3 Vectors の「低頻度アクセス向け」という位置づけは、この構造なら個人〜中規模用途で全く問題にならない。

### 4. KB Retrieve は「生チャンク」を返さない（マネージドの見えない付加価値）

同一インデックスへの同一クエリで、③の直叩きは 228〜690字の生チャンク5件、②の KB Retrieve は**最大 8,120字に結合・拡張されたチャンク**を返した。結果として ② は入力トークン 3.3倍・生成 2倍遅・ただし回答はより深い。**品質(②) vs 軽さ(③) のトレードオフ**が実測で確認できた。この拡張の正確な仕組み（親チャンク返却か隣接結合か）は要追加検証。

### 5. 自前化するほど IAM が太る

① は `bedrock:Retrieve*` + 生成モデルで済むが、③ は埋め込みモデルの `InvokeModel` と `s3vectors:QueryVectors` に加え、**`returnMetadata=true` には `s3vectors:GetVectors` も必要**（エラーで実測発見。ドキュメントからは読み取りにくい)。マネージドを剥がすことは、権限管理の責任を引き受けることでもある。

### 6. スコアの意味が方式で違う

KB Retrieve の score（大きいほど近い・正規化済み）と S3 Vectors の distance（cosine・小さいほど近い）は体系が別物。閾値設計を移植するときは要注意。

## 使い分け指針（個人ラボの結論）

- **単発の質問応答・とにかく速く作る** → ①
- **マルチターン対話・プロンプトを制御したい** → ②（検索品質は KB に任せつつ制御を握る、バランス型）
- **RAG の内部を理解する・検索テクニックを自作する（クエリ書き換え、リランキング等）** → ③

## 再現方法

```bash
# ① managed
python chat.py --profile bedrock-lab --rag

# ② 検索のみ managed
python mini_rag.py --profile bedrock-lab --search kb --verbose

# ③ フル自前
python mini_rag.py --profile bedrock-lab --search vectors --verbose
```

計測ログは `client/logs/minirag_*.jsonl`（timings / usage / hits を記録)。
